I did some basic profiling while the map was loaded as this seems to be the bottleneck for me.

This showed me that close to 11% are used in org.mapsforge.android.maps.MercatorProjection.latitudeToPixelY(double, byte)

From thilo.mu...@gmail.com on February 16, 2011 10:22:02

How exactly did you do the profiling? I have measured the library several times and the MercatorProjection class was never a bottleneck. I doubt that really 11% or more of the time are spent in the latitudeToPixelY() method. Unfortunately the Android method tracing is not very accurate sometimes, so you can not rely on its numbers.

A lookup table is not a good solution. First, we want to calculate the projection on all zoom levels from 0 to 25 with a very high precision. For that we would need a huge lookup table that consumes too much RAM. As RAM is very limitted (Android gives each process only 16-24 MB), this is really a problem. Second, having readable source code is important as well. PI / 180 is an understandable formula, 0.017453293 is not. And third, i assume that every modern compiler will convert the expression to its numerical result while optimizing the source code.

If we want to improve the performance, there are better places to start with. For example, new devices will have multi core CPUs, how can we make use of them (see issue #73 )?

"Premature optimization is the root of all evil." (Donald Knuth)

I indeed used android's method profiling. So if this is not very accurate probably the 11% are wrong as well.

I did not realize from quickly reading the code that the precision is that important. If that is the case, I agree that lookup tables don't work.

On the readability, there is some middle ground. You could pre-calculate PI/180 in an a static field. So it would remain readable. A quick look into the decompiled code shows

double d1 = Math.sin(3.141592653589793D * paramDouble / 180.0D);

and

double d4 = Math.log(d2 / d3) / 12.566370614359172D;

So you are partially right. Android optimized 4*PI, but does not optimize PI/180. However if the whole profiling was wrong, the optimization might not be necessary.

Regarding multicore support I agree, however this won't help obviously on old devices.

From thilo.mu...@gmail.com on February 17, 2011 03:30:13

That's an interesting finding. I just used "javap -c" to look at the byte code of the latitudeToPixelY method. The corresponding line of code is:

double sinLatitude = Math.sin(latitude * Math.PI / 180);

It looks like the compiler is not as clever as i thought. It first multiplies (latitude * Math.PI) and then divides it by 180, resulting in two arithmetical operations.

Now i modified the source code and added two brackets:
double sinLatitude = Math.sin(latitude * (Math.PI / 180));
The result is a constant of 0.017453292519943295 in the bytecode and the division is gone. Only a single arithmetical operation for the same expression!

To cut a long story short: we can simply add (...) and save one division per latitudeToPixelY() call. Besides that we should fill a bug report to let the developers know that their Java compiler sucks.
