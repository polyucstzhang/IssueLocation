I have just managed to reproduce this. It is a deadlock situation where one thread first acquires the MapViewPosition then the FrameBuffer, while another thread acquires the same objects in the opposite order. I have changed the title of this issue, so that it is clear that this is not a random ANR.

Ordered acquisition of the objects should fix this. I will try to publish a fix today.

From LudwigBr...@gmail.com on December 17, 2012 00:42:52

IMHO there are two options to fix it:
1.) keep the locking the same, but ensure ordering, so that the FrameBuffer is always locked before the MapViewPosition. That way, no deadlock can occur. However, this requires that all methods write change the MapViewPosition follow this protocol and from looking at the code I thought this a bit messy.
2.) move the access to the MapViewPosition out of synchronization. Read access to the MVP is synchronized, so there is no risk that a inconsistent MVP is retrieved, but it is possible that the MVP position gets changed while the FrameBuffer is redrawn. However, any effect would be overdrawn in the next redraw operation that gets scheduled by the MVP change. I have tested this and I have not been able to spot any visual artifacts.

So for my fix I have opted for the second solution, which essentially a very small change, with little risk.

Attached is a patch, but I have put a fix for this issue on my clone, the fix is just a single commit to the current head of master, you can find it here: https://code.google.com/r/ludwigbrinckmann-mapsforge/source/list?name=i358 Please give it a try and report any problems (or lack thereof) back.
