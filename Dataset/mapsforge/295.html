From thilo.mu...@gmail.com on April 15, 2012 06:15:05

This sounds like a misunderstanding. The bbox parameter should only be used in cases where the input file does not contain a bounding box definition or to overwrite invalid values in the input file.

If you are using a larger input file (e.g. Austria), you cannot use the bbox parameter to extract a smaller region from it. However, you can do this with the Osmosis tool, see http://wiki.openstreetmap.org/wiki/Osmosis#Extracting_bounding_boxes for a description.

@Jürgen: please correct me if i am wrong. Otherwise close this issue as invalid and consider adding this important information to the description of the bbox parameter on the GettingStartedMapWriter wiki page to avoid this misunderstanding in the future.

From jjbross@gmail.com on April 23, 2012 11:38:15

You can(!) use the bbox parameter to restrict the map to a certain area. In the following a short explanation:

The map file is tile based. A single file contains multiple tiles which always cover a rectangular area. This rectangular area is combuted by means of the bbox (either read from the
OSM data file or from the parameter). If you specify a smaller bbox than contained in the OSM data, only the tiles (and the relevant data) covered by this bbox will be written to the file. Specifying a larger bbox is possible, but makes no sense, as empty tiles will be generated.

A way read from the input data that does not cross any of the tiles defined by the bbox, is discarded. A way may only be partly covered by the tile area defined by the bbox. Parts that are not covered by any tile, are discarded, the other parts are included.

As comment6 indicates, the problem initially described is not related to the use of the bbox parameter. The way in question was partly cut out by the osmosis call commented with "#cut out osm-data from austria.osm". The map writer got incomplete input (some way nodes were missing) and thus completely discarded the way. This happens with every way which refers to missing way nodes and thus most often occurs add boundaries of the input data.
