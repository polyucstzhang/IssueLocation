Validation for custom search URL fails #80

Closed
eljoest opened this Issue Mar 5, 2014 · 17 comments

Labels

None yet

Milestone

Assignee

srowen

eljoest commented Mar 5, 2014

The validation code for the custom search URL in PreferencesFragment seems to pass the user input to the URL class without removing the placeholders (%s etc).
Unfortunately this seems to yield a syntax error on KitKat as %s is not a valid escape code and thus prevents the use of a custom search URL.
If I replace the % with %25 (escaped %) the URL is accepted but the Barcode contents does not get inserted.
Test case: KitKat on Galaxy S4
URL rejected: https://www.google.com/search?q=%s
URL accepted but not working: https://www.google.com/search?q=%25s

Inserting the rejected URL directly into the preferences file (as root) works as expected.

Possible solution: Substitution of the placeholders with an empty string prior to the validation.

eljoest commented Mar 5, 2014

I checked fillInCustomSearchURL and noticed an additional issue: The code will break escaped characters in the range 0xf0 to 0xff. While of no concern for the validation here it's probably not quite right for non-ascii use in fillInCustomSearchURL. Additionally any Barcode that contains characters in that range will have these occurrences replaced, too. Literal % in the barcode are protected due to the URLencode of the string.

Im not quite sure if those characters are actually needed, but it might spell trouble within UTF-8 sequences (URL as well as barcode) and binary content. Maybe the replacement code should be reworked to allow for %%-style escaping of these sequences for the URL and do the replacements in one go with a java.util.regex.Matcher and its find+group+appendReplacement.
What do you think, is it too exotic a use case to even worry about?

Owner
srowen commented Mar 6, 2014
Hm that is quite true -- was already true I suppose, since "%f" is a placeholder. It was supposed to be a clever imitation of printf placeholders but that's an unfortunate conflict with URL escapes, yes. I suppose the lesser of two evils is to not replace occurrences like "%f[0-9a-f]". That causes a different kind of problem, but probably much less common. As to the second issue, I think it is solved by replacing %s last, since the other two can never contain values with an escape sequence.
